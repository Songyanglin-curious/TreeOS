我先把回合协议讲成一个“精确定义”，再讨论它为什么成立、边界在哪里、以及你可能需要的增强项（仍保持最小，不把它工程化）。

## 1) 回合协议的严格定义：它不是流程，是“每回合的状态机”

一个回合的目标不是“把问题解决”，而是：

* **把系统状态从 S(t) 推进到 S(t+1)**
* 同时保证：S(t+1) 可接续、可回溯、可分支

因此回合协议的本质是三件事：

### Expand：产生候选（Candidate Set）

输入：当前上下文（主题+已有沉淀+约束）
输出：候选集合 C = {c1, c2, …}
候选可以是：视角、假设、反例、选项、版本、下一步问题、练习题、设计草图

最低要求：

* 候选要“同一维度”或“标注维度”，否则无法比较
* 候选数量不追求多，追求覆盖关键维度（后面讲“扫全”策略）

### Select：选择推进（Active Focus）

输入：候选集合 C + 选择准则 P
输出：焦点集合 F（通常 1–2 个）+ 其余进入 Parking Lot

Select 的本质是： **承诺本回合只推进 F，其他不丢但暂停** 。
它解决的是“横向铺开导致无法累积”的系统性问题。

### Crystallize：固化可接续状态（Next State Artifact）

输入：本回合推进结果（围绕 F 的新增理解/产出）
输出：最小沉淀 A（可作为下一回合输入）+ 更新后的 Parking Lot

Crystallize 的“最低合格线”是：

* 下一回合不需要回忆整段对话，也能继续推进
* 可以明确本回合“推进了什么/没推进什么”

这就是它为什么是协议：它规定了**候选、焦点、沉淀**三种对象的生成与转移。

## 2) 为什么这三步是必要且最小的

你现在的痛点可以精确映射：

* 没有 Expand：你只是在沿着单一路径“搜答案”，容易盲区、容易被默认假设绑架
* 没有 Select：你会进入“候选爆炸”，讨论热闹但无法推进
* 没有 Crystallize：你每次都从零开始，越聊越深但资产不增长

所以三步里任何一步拿掉，都会产生一种典型失控：

* 去掉 Expand → 变成僵硬收敛/单点钻牛角尖
* 去掉 Select → 变成摊大饼/跑偏
* 去掉 Crystallize → 变成空转/无沉淀

这也是它“正交”的意义：每一步解决一种不同的失败模式。

## 3) “Select=1–2”会不会伤害全面性：关键在“候选池如何扫全”

你担心全面性，是合理的，但问题不在 1–2，而在 Expand 的产出质量。

要保证全面性，你需要一个“扫全但不铺开”的 Expand 策略：
Expand 不追求“列很多”，而追求“覆盖关键维度”。

一个很实用的最小策略是：Expand 输出必须包含三类候选（每类 1–2 个即可）：

* **解释类** ：我可能如何理解这个问题（视角/框架）
* **不确定类** ：决定走向的关键变量/假设
* **反证类** ：可能推翻当前理解的反例/失败模式

这样你每轮只选 1–2 个深挖，跨回合仍能覆盖全局。

## 4) 回合协议的边界：它不解决“评价标准”，只解决“推进与累积”

回合协议本身不告诉你“好坏”“对错”“是否交付”，它只保证两件事：

* 不跑偏（因为有 Select）
* 可累积（因为有 Crystallize）

评价标准来自模式插件：

* 思考：张力是否清晰、变量是否被识别
* 学习：是否能做对题/迁移应用
* 创作：是否达到表达效果与一致性
* 工程：是否可验收、可运行、风险可控

所以你不用把回合协议写得像工程规范；它只是一条“每回合必须留下可接续状态”的操作系统内核。

## 5) 你可能需要的“最小增强”，但不破坏最小性

如果你要把它从“好想法”变成“稳定可运行”，最小增强通常只有两个：

### 增强 A：Select 的准则固定化（避免凭感觉挑）

只需要二选一（不需要复杂评分）：

* 信息增益优先：选最不确定、最可能推翻理解的候选
* 影响优先：选错了代价最大/收益最大的候选

### 增强 B：Crystallize 的格式固定化（你已经在做）

你现在用 3 行沉淀 + Parking Lot，就是最小增强。

## 6) 结论：你现在的回合协议已经很接近“可作为 TreeOS 内核”

但要让它真正成为“基础设施”，你需要明确两句话（写进你的宪章里）：

* **回合的成功标准不是解决问题，而是产生可接续状态**
* **全面性靠跨回合覆盖，单回合只允许推进 1–2 个焦点**

这两句话会极大减少你对“是不是太不全面/是不是没结论”的焦虑，因为你的系统目的从“结论”转移到了“可持续推进”。

Crystallize（本轮沉淀）

* 当前表述 vN：TreeOS 的最小回合协议本质是“状态机”：候选集合→焦点集合→可接续沉淀。
* 关键张力/变量：单回合推进效率 vs 单回合全面铺开；“解决问题”目标 vs “产生可接续状态”目标。
* 下一步观察/试探：为 Expand 定一个“扫全但不铺开”的最小输出约束（3类候选：解释/不确定/反证），并为 Select 固定一个准则（信息增益优先或影响优先）。

Parking Lot

* 回合协议是否需要显式定义“状态对象”字段（Candidate/Focus/Artifact/ParkingLot）以便未来编码。
* 不同模式下 Crystallize 的产物协议如何统一命名与存储（3行、卡片、导演指令、规格树）。
* 是否需要“分支/回滚”语义（当 Select 选错方向时如何低成本回退）。
