# TreeOS核心

## 人机协作的认识流水线(Thinking Pipeline)

本质是：**元认知系统**

TreeOS 是一套人机协作的思考工作流：用统一的回合协议（Expand/Select/Crystallize）驱动不同模式插件（思考/学习/创作/工程），把模糊输入转化为可接续、可复用、必要时可验收的资产。


* 不同模式可以共用一个最小骨架：Expand（扩展候选）→ Select（只推进 1–2 个）→ Crystallize（留下可接续状态）。
* 你当前最大缺口不在“收敛能力”，而在“沉淀协议（Library）+ 上游开图（Expander）”。


## 你现在已确定的关键结论（不等于最终结论）

* ThinkingNode 是 TreeOS 的一个模块：适用于“需求收敛/可验收规格化”（工程强收敛）。
* TreeOS 本体不是“树/思考原子/结构化”本身，而是“端到端流水线 + 模式切换 + 可累积沉淀”。
* 不同模式可以共用一个最小骨架：Expand（扩展候选）→ Select（只推进 1–2 个）→ Crystallize（留下可接续状态）。
* 你当前最大缺口不在“收敛能力”，而在“沉淀协议（Library）+ 上游开图（Expander）”。

## 3) TreeOS 的共用骨架（回合协议）

每一轮必须完成：

* Expand：生成候选池（视角/假设/反例/选项/版本）
* Select：只选 1–2 个继续（其他进 Parking Lot）
* Crystallize：输出最小沉淀单位（不同模式固化对象不同）

“收敛”在这里不是收敛成结论，而是收敛成：本轮推进方向 + 本轮留下的东西。

## 4) 模块地图（TreeOS 0.1 最小模块集）

优先级从高到低：

1. Library（沉淀库协议）：确保每轮都有可接续状态
2. Expander（思考开图）：产出张力/变量/探索方向（不出方案）
3. Mode Switcher（模式选择器）：决定本轮用思考/学习/创作/工程
4. ThinkingNode（你已有）：需求收敛为可验收规格树

   （Builder / Coder / Deploy / Trainer 可后置）

## 5) 不同模式的“固化对象”（Crystallize 的产物协议）

* 思考：3 行沉淀
  * 表述 vN
  * 关键张力/变量
  * 下一步观察
* 学习：可复练卡片
  * 一句话概念
  * 易错点
  * 触发题+要点
* 创作：导演指令
  * 本段效果
  * 禁止项
  * 下一版目标
* 工程：规格树/设计包/提交记录（你已有一部分）

## 6) Select=1–2 不会损失全面性的前提

* Expand 先扫全形成候选池
* 未选项进入 Parking Lot
* Select 依据“信息增益最大”或“影响最大”挑 1–2 个推进
* 全面性靠跨回合覆盖，而非单回合铺开

## 7) 你现在的“下一步”（最小可执行）

做 3 次手动跑通（不写代码）：

* 主题 A（思考类）：只跑 Expander + 3 行沉淀
* 主题 B（学习类）：只跑 Trainer + 卡片
* 主题 C（工程类）：从 Expander 产出探索方向后进入 ThinkingNode

每次结束都必须写入 Library（对应沉淀对象）。跑完你自然知道 TreeOS 0.1 的模块边界与缺口。


## TreeOS 的端到端链路（最小闭环）

0. 输入：一句话/一个困扰/一个想法
1. 思考开图（Expander）：扩展问题空间，产出张力/变量/探索方向
2. 学习补洞（Trainer，可选）：补齐必要知识与能力，靠练习验证
3. 需求收敛（ThinkingNode）：把目标变成 what/why/boundary/acceptance 的可验收规格树
4. 设计与实现（Builder）：模块/接口/数据结构/测试策略（可以先轻后重）
5. 生成与迭代（Coder/Deploy）：生成代码→运行→测试→修复→部署
6. 复盘沉淀（Library）：沉淀模板、反模式、检查清单、决策记录（成为下次的上下文）

这条链路的关键不是“每次都走完”，而是：每一步都有独立产物，且下一步只吃上一步的产物。

## TreeOS正交基石

> 系统级基石更像“**协议**”和“**最小产物**”。

TreeOS 的正交基石优先是三类：

A. 模式切换协议（Mode Switcher）

输入同一个主题，允许切换：思考/学习/需求/实现。

切换依据不是感觉，而是“目标与约束”。

B. 最小产物协议（Artifacts）

每个阶段产物必须极简、可复用、可作为下一步输入。

例如：

* 思考产物：张力对 + 关键变量 + 下一步探索
* 学习产物：可复练卡片
* 需求产物：ThinkingNode 树
* 实现产物：模块草图 + 接口契约 + 测试要点
* 复盘产物：反模式/检查清单/可复用片段

C. 低摩擦闭环（Loop）

任何阶段都能以“最小步骤”回到可推进状态，而不是卡死在完美主义。

这三个才是“系统基石”；具体模块（ThinkingNode/expander/trainer）只是挂载其上的实现。

## 子模块

### 需求收敛(ThinkingNode)

> **TreeOS 的唯一目标是实现思考收敛**

- AI 负责发散：补全隐含信息，提供候选方案
- 人类负责收敛：确认节点有效性，决定结构走向

约束和组织问题的最小单元 我称之为 [思考原子](./模块/需求收敛/基石/思考原子.md) ,它以 **是什么**、**为什么**、**边界**、**验收标准** 四个单元来回答和约束一个问题,这四点 **正交**且**完备**；**思考单元**具有**自相似性**和**递归特性**;一个问题由一堆子问题有层级地组合起来的树状结构或流程图，对每一个拆解出来的子问题，再次套用这个**思考原子**并回答四个单元。

#### 工作流总览（四步循环）

1. 选定一个“当前问题”（一个节点）
2. 让 AI 生成该节点的  [ThinkingAtom](./references/ThinkingAtom.ts)（what/why/boundary/acceptance）
3. 你做“收敛审查”（改到可接受为止）
4. 判断要不要拆：
   * 可直接验收 → 叶子节点
   * 不可直接验收 → 让 AI 给 children（子节点），进入下一轮
