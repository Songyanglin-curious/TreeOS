
# Ascend 手动操作流程 0.2

## 一、流程目标

每一轮只做一件事：

**推进当前一个节点，并把结果写回你的宏观树。**

不是每轮解决整个问题。
不是每轮重建整棵树。
不是每轮都处理所有分支。

## 二、每轮固定流程

### 第 0 步：从宏观树里确定当前节点

你先看自己维护的 js 对象，只做 4 个确认：

1. 总问题是什么
2. 当前主线节点是什么
3. 本轮只推进哪个节点
4. 还有哪些已知分支先不动

这里不用你每次现写长文本。
本质上只是：

**从树里读出“本轮我要处理谁”。**

所以这一步不是“写作动作”，而是“定位动作”。

### 第 1 步：把当前节点丢给 AI 做扩展

这一轮只让 AI 围绕当前节点展开。

目标不是要最终答案，而是看清以下三种情况之一：

1. 这个节点讨论对象还没稳定
2. 对象已经稳定，但候选方向还不够清楚
3. 候选方向已经有了，可以准备选择

这一轮扩展的产出，核心只看三类东西：

* 当前节点更清楚了什么
* 是否冒出了新分支
* 是否出现了会影响其他节点的变化

### 第 2 步：你做人为选择

你看扩展结果后，只做选择，不做长篇总结。

你这一轮通常只需要定 3 件事：

1. 当前节点暂时采纳什么判断
2. 哪些新东西要记成分支
3. 是否需要标记某些节点待回扫

这一步的本质就是：

**把 AI 给出的可能性，收成当前可推进的一条线。**

### 第 3 步：生成本轮沉淀

本轮沉淀不是全局总结，而是：

**当前节点的一次局部冻结。**

沉淀只需要服务一件事：

**让下一轮能接着推进。**

所以你这一轮沉淀只需要包含这几类信息：

1. 本轮推进的是哪个节点
2. 本轮新增了什么认识
3. 当前冻结结论是什么
4. 新增了哪些分支
5. 有没有待回扫节点
6. 下一轮默认从哪继续

注意：
沉淀不是用来记录所有讨论过程的。
只记最后有效结果。

### 第 4 步：写回宏观树

你把本轮沉淀的结果更新进你的 js 对象。

更新动作通常只有这几类：

1. 更新当前节点状态
2. 更新当前节点结论
3. 新增分支节点
4. 给某些节点加“待回扫”标记
5. 确认当前主线是否变化

这一步很关键。

因为从现在开始：

* 聊天记录不是主记录
* 沉淀文本也不是主结构
* **你的 js 树对象才是主结构**

### 第 5 步：决定下一轮入口

每轮最后只定一件事：

**下一轮先推进谁。**

通常只有三种情况：

1. 继续当前节点
2. 切到某个保留分支
3. 先做一次回扫

到这里一轮就结束。

## 三、这一套流程里，各部分分别算什么

为了防止后面又混掉，我给你压成最短定义。

### 1. 宏观树是什么

宏观树是：

**问题结构的主记录。**

它负责保存：

* 总问题
* 当前主线
* 各节点关系
* 分支状态
* 待回扫标记

你用 js 对象维护它，这个方向是对的。

### 2. 每轮扩展是什么

扩展是：

**对当前节点的局部展开。**

不是展开全问题。

### 3. Select 是什么

Select 是：

**你对本轮推进方向做人工拍板。**

它决定：

* 当前采纳什么
* 什么挂为分支
* 什么标记待回扫

### 4. 沉淀是什么

沉淀是：

**当前节点本轮推进后的局部冻结记录。**

不是全局总结。

### 5. 写回是什么

写回是：

**把沉淀结果真正更新到宏观树。**

沉淀是文本结果。
树更新才是结构结果。

### 6. 回扫是什么

回扫是：

**当某个上位理解变了之后，去检查受影响节点。**

不是重新跑全树。

## 四、什么时候记分支

只在下面这些情况下记分支：

1. 这个问题已经值得单独处理
2. 它本轮不推进，但后面明显要处理
3. 它会影响主线判断
4. 它是一个未选中的重要候选方向
5. 它反复打断当前主线

除此之外，不要什么都挂树上。

原则就是：

**不记会丢的重要问题才记。**

## 五、什么时候标记回扫

只在下面这些情况下标记待回扫：

1. 上位定义变了
2. 核心对象变了
3. 核心边界变了
4. 当前结论推翻了旧假设
5. 其他节点明显依赖旧理解

注意是 **标记待回扫** ，不是立刻大范围重做。

## 六、你真实执行时的最简版本

如果你想极简执行，那你每轮实际只做下面这 6 件事：

1. 从 js 树里选当前节点
2. 让 AI 只扩展这个节点
3. 你做选择
4. 让 AI 产出这一节点的沉淀
5. 你把结果写回 js 树
6. 定下一轮节点

这就是完整闭环。

## 七、和你原来三步法的关系

你原来是：

**Expand → Select → Crystallize**

这个没有废掉。
只是现在要补上前后两个结构动作。

所以新的真实操作形态其实是：

**节点定位 → Expand → Select → Crystallize → 写回树 → 决定下一轮**

也可以压缩理解成：

**树上选点，围点推进，结果回树。**

这句话你后面拿来当流程总纲就够了。

## 八、当前最适合你的使用方式

你现在最适合的不是再继续抽象，而是：

1. 先维护一个最小 js 树对象
2. 每轮只选一个节点推进
3. 聊完后把结果写回树
4. 用那个“三层任务下发 / 多天回送 / 归档统计”需求去试跑

这样你能很快验证两件事：

* 这个流程会不会太重
* 分支、沉淀、回扫到底够不够用

最后给你一个最短版，后面你自己看这个就能想起来：

**每轮先从树里选一个当前节点，只围绕它扩展；你做人为选择；把结果沉淀成该节点的局部冻结；再写回树，并决定下一轮推进入口。**
