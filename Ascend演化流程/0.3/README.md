# Ascend

Ascend 是一个借助 AI 放大个人生产力、帮助人持续升维的人机协作系统。

在当前阶段，Ascend 聚焦解决的不是“自动生成答案”，也不是“直接产出代码”，而是：

**让一个复杂、易分叉、会反复改写上位理解的问题，能够在人与 AI 的协作中持续推进，而不丢结构、不丢主线、不因上下文膨胀而失控。**

因此，Ascend 当前更准确的落点是：

**一个由微观循环驱动、由宏观树承载、允许分支保留与局部回扫的动态问题推进架构。**

## 目录说明

### `spec/`

Ascend 的系统级定义文档。

这里保存的是：

- Ascend 是什么
- 为什么存在
- 当前阶段解决什么
- 总体架构是什么
- 微观循环、宏观树、分支、Freeze、回扫分别是什么
- 哪些内容已经冻结为当前共识
- 哪些内容当前不进入

这里不放运行时提示词。

### `prompts/`

Ascend 的运行时短提示词。

这里保存的是可直接投喂 AI 的模块提示词，包括：

- Ascend Core
- Ascend Executor
- Ascend Select
- Ascend Freeze
- Ascend Reviewer
- Ascend Tree

这些文件只负责运行，不承担系统设计说明。

### `archive/`

历史材料与旧版本归档。

这里保存：

- 旧版提示词
- 旧笔记
- 已被新结构替代但仍有参考价值的材料

归档的作用不是继续使用，而是保留演化痕迹，避免污染主目录。

## Ascend 的当前结构

Ascend 当前采用两层耦合结构：

### 1. 宏观层：问题树

用于承载整个问题空间的结构，负责：

- 保存当前主线
- 保存分支与挂起项
- 支持后续切换与局部回扫

### 2. 微观层：推进循环

用于推进当前焦点问题。

当前最小循环为：

**Expand → 人工 Select → Freeze**

其中：

- Expand：围绕当前节点推进一轮
- Select：由人做主线选择
- Freeze：把本轮结果冻结成可接续状态

两层关系不是分离的，而是：

- 树提供当前推进位置
- 微观循环推进该位置
- 推进结果写回树
- 写回结果可能改变主线、产生分支、触发回扫

1. `spec/00-overview.md`
2. `spec/01-system-definition.md`
3. `spec/03-architecture.md`
4. `spec/04-micro-loop.md`
5. `spec/05-macro-tree.md`
6. `prompts/` 下各模块提示词

如果你想快速理解系统，只需先看：

- `README.md`
- `spec/00-overview.md`

- `spec/`
- 再看对应 `prompts/`

## 当前边界

Ascend 当前阶段只聚焦复杂问题推进架构本身。

当前不进入：

- UI 设计
- 数据库存储
- 工程实现细节
- 自动化编排细节
- 产品化交互细节

这些内容只有在系统级定义足够稳定后才进入。
