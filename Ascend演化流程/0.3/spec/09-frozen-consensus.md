# 09 Frozen Consensus

## 1. 本章目的

本章用于记录 Ascend 当前阶段已经冻结的核心共识。

这里的“冻结”指的是：

- 当前已经明确成立
- 后续可以继续使用
- 不再作为日常讨论中的开放问题反复摇摆

本章不是为了证明这些共识为什么成立，而是为了：

**为后续架构、模块、提示词与工程化讨论提供稳定前提。**

因此，本章更接近“当前系统母法中的已确认部分”，而不是分析文。

---

## 2. 关于 Ascend 的系统定位

### 2.1 Ascend 当前不是单一工具

Ascend 不是一个单点工具，不只是一个页面、一个树组件、一个提示词包，也不只是“需求收敛器”。

### 2.2 Ascend 当前更准确的定位

Ascend 当前更准确的定位是：

**一个借助 AI 放大个人生产力、帮助人持续升维的人机协作系统。**

### 2.3 Ascend 当前阶段的具体落点

虽然 Ascend 的长期方向更大，但当前阶段的具体落点已经冻结为：

**复杂问题推进架构。**

当前优先解决的不是 UI、数据库、自动化或产品包装，而是：

**让一个复杂、易分叉、会反复改写上位理解的问题，能够在人与 AI 的协作中持续推进，而不丢结构、不丢主线、不因上下文膨胀而失控。**

---

## 3. 关于总体结构的冻结共识

### 3.1 Ascend 当前采用双层耦合架构

已冻结为：

- 宏观层：问题树
- 微观层：推进循环

两层不是孤立并列，而是耦合关系。

### 3.2 宏观层的定位

宏观层的定位已经冻结为：

**问题空间的结构化地图。**

它负责承载长期结构，而不是记录完整思维过程。

### 3.3 微观层的定位

微观层的定位已经冻结为：

**围绕当前节点完成一轮最小推进的机制。**

它负责当前轮推进，而不是维护全局结构。

### 3.4 两层关系

已冻结为：

- 树提供当前推进位置
- 微观循环围绕该位置推进一轮
- 推进结果写回树
- 写回结果可能继续压实当前节点、形成分支、改写上位理解或触发局部回扫

因此，Ascend 不是静态拆树，而是：

**用微观循环驱动宏观问题树持续演化。**

---

## 4. 关于微观循环的冻结共识

### 4.1 当前最小循环已冻结

当前最小循环已冻结为：

**Expand → 人工 Select → Freeze**

### 4.2 Select 必须存在，而且由人执行

这一点已冻结，不再摇摆。

Select 的存在不是附属动作，而是 Ascend 的关键控制点。

### 4.3 Freeze 是正式名称

在当前正式命名体系中，第三步统一使用：

**Freeze**

不再以 Crystallize 作为正式运行时主称呼。

### 4.4 微观循环的目标

微观循环的目标不是一次性解决问题，而是：

**让复杂问题在不完整、不稳定、可改写的状态下，仍然能推进一轮。**

---

## 5. 关于 Expand 的冻结共识

### 5.1 Expand 采用 A / B / C 三态动态输出

已冻结为：

- A：问题对象未稳定
- B：对象已稳定，但候选池未形成
- C：候选池已形成，可进入 Select

### 5.2 Expand 不是固定模板输出

它必须根据当前成熟度动态变化，而不是机械套模板。

### 5.3 Expand 的任务不是直接收敛

Expand 的作用是推进当前节点，使其更清楚、更可判断、更接近可选择状态。

### 5.4 未形成真正可比候选前，不进入 Select

这条已经冻结为全局原则。

---

## 6. 关于 Select 的冻结共识

### 6.1 Select 由人来做

这条已冻结，不作为开放问题继续讨论。

### 6.2 Select 的核心作用

已冻结为：

- 选择当前主线
- 确认暂不推进的保留分支
- 阻止 AI 自动并行扩张失控
- 维持问题推进中的主观控制权

### 6.3 Select 不是可省略步骤

Ascend 不接受“AI 自己展开、自己判断、自己拍板”构成完整推进循环。

---

## 7. 关于 Freeze 的冻结共识

### 7.1 Freeze 的作用

已冻结为：

**将本轮推进结果冻结为可接续状态。**

### 7.2 Freeze 不是总结器

Freeze 不是：

- 最终结论
- 完整总结
- 全过程归档
- 终局定稿

### 7.3 Freeze 只服务接续

Freeze 的核心目的已经冻结为：

- 边界更清
- 当前可继续
- 轮次之间可接续

### 7.4 Ascend 当前采用弱收敛

这一点已冻结。

Freeze 服务的是弱收敛，而不是 ThinkingNode 式的强收敛。

---

## 8. 关于宏观树的冻结共识

### 8.1 宏观树是长期承载结构

已冻结为：

**树负责保存复杂性，但不直接执行复杂性。**

### 8.2 树不是执行并行器

这是当前架构中的关键冻结修正。

已冻结为：

**认知结构上是树，执行节奏上默认单主线推进。**

### 8.3 宏观树的最小职责

当前已冻结的最小职责包括：

- 承载问题结构
- 定位当前主线
- 支持后续校正

### 8.4 树不记录全过程分析

树保存结构状态，不保存完整思维过程。

---

## 9. 关于分支机制的冻结共识

### 9.1 允许多分支存在

这一点已冻结。

Ascend 承认复杂问题推进中天然会长出多个分支。

### 9.2 分支存在不等于并行推进

这一点已冻结，而且是架构级原则。

### 9.3 每轮默认只推进一个主分支

这一点已冻结。

### 9.4 其他分支应被保留

其他分支不应丢失，而应进入分支池、挂起状态或宏观树的适当位置。

### 9.5 分支何时升格

当前已冻结的判断方向是：

**只有当一个新出现的问题具备独立问题价值时，才升格为分支。**

---

## 10. 关于回扫的冻结共识

### 10.1 Ascend 允许局部回扫

这一点已冻结。

### 10.2 回扫的触发源

当前已冻结为以下几类上位变化：

- 上位定义变化
- 核心对象变化
- 主线判断变化
- 关键边界变化

### 10.3 回扫不是全树重做

这一点已冻结，而且非常关键。

回扫的定位已经冻结为：

**只对受影响部分进行局部重新检查与一致性校正。**

### 10.4 回扫的目标

回扫服务的是：

- 恢复结构一致性
- 恢复主线可继续性

而不是重做全部分析。

---

## 11. 关于推进节奏的冻结共识

### 11.1 默认单主线推进

这一点贯穿微观循环、宏观树、分支机制，已冻结为全局原则。

### 11.2 保留复杂性，但不同时执行复杂性

这一点已冻结为 Ascend 当前最重要的架构取向之一。

### 11.3 宁可保留，也不要丢失

对于重要但当前不推进的内容，默认取向是轻量保留，而不是直接删除。

### 11.4 宁可继续压实，也不要提前建模

这条也已冻结为当前系统推进原则之一。

---

## 12. 关于当前阶段边界的冻结共识

### 12.1 当前阶段只聚焦复杂问题推进架构

这一点已冻结。

### 12.2 当前不进入以下内容

已冻结为当前不进入：

- UI 设计
- 数据库存储设计
- 工程实现细节
- 自动化编排细节
- 产品化包装细节

### 12.3 当前不把临时提示词技巧上升为系统定义

运行技巧、偶然调参经验、单轮临时优化，不直接进入 Spec 主体。

---

## 13. 关于正式命名的冻结共识

当前正式命名体系已冻结为：

- Ascend Core
- Ascend Executor
- Ascend Select
- Ascend Freeze
- Ascend Reviewer
- Ascend Tree

对应命令已冻结为：

- `/go`
- `/select`
- `/freeze`
- `/review`
- `/tree`

旧称呼如：

- Crystallize
- Tree Updater
- Select Helper

可以作为历史兼容理解存在，但不再作为正式主称呼。

---

## 14. 本章结论

截至当前阶段，Ascend 已冻结的核心共识可以浓缩为以下几句话：

- Ascend 当前落点是复杂问题推进架构
- 它采用宏观树 + 微观循环的双层耦合结构
- 最小微观循环是 Expand → 人工 Select → Freeze
- Expand 采用 A / B / C 三态动态输出
- Select 必须存在，而且由人执行
- Freeze 服务弱收敛与下一轮接续
- 树负责保存复杂性，执行默认仍是单主线
- 允许多分支存在，但不默认并行推进
- 上位变化时允许局部回扫，但不做全树重来

本章的作用不是解释这些共识为什么成立，而是：

**把当前已经不想再反复摇摆的前提集中冻结下来，作为后续继续推进的共同基础。**
