
# 冀北超高压流程 V1 对象与规则清单

## 一、对象清单

---

## 1. 总单

### 对象定位

A 发起的顶层安排对象。

### 谁创建

A 创建。

### 它解决什么问题

* 表达一次顶层工作安排
* 作为 B 后续建任务的来源
* 让 A 能从总单维度看整体推进

### 最小字段

* id
* 标题
* 内容说明
* 创建人 A
* 创建时间

### 当前边界

V1 里总单只负责顶层来源，不承担复杂流程状态，不承担细粒度审批。

---

## 2. 任务

### 对象定位

B 基于总单创建，并派给某个 C 的主业务对象。

### 谁创建

B 创建。

### 它解决什么问题

* 成为 C 每天填报的主轴
* 承载责任归属
* 承载时间范围
* 成为“应报范围”的计算基础

### 最小字段

* id
* 所属总单 id
* 所属 B
* 责任 C
* 任务名称
* 开始日期
* 结束日期
* 状态

### 当前边界

V1 默认：

* 一个任务只派给一个 C
* 不处理一任务多人
* 不处理一人多任务冲突

---

## 3. 任务日报

### 对象定位

C 围绕某个任务、某一天提交的一条日报实例。

### 谁创建/修改

* C 首次填报时产生
* 被 B 打回后，仍由 C 在原实例上修改并重提

### 它解决什么问题

* 记录 C 某天对某任务的实际填报
* 承载日报内容
* 承载 B 打回后的修正闭环

### 最小字段

* id
* 所属任务 id
* 所属日期
* 填报人 C
* 日报内容
* 状态
* 提交时间
* 打回原因

### 当前边界

V1 明确：

* 单任务 + 单日期 = 单日报实例
* 不允许同一天并列多条日报
* 修改重提不新建并列记录

---

## 4. B 当日上送实例

### 对象定位

B 在某一天向 A 的一次整份上送实例。

### 谁创建/提交

B 发起并提交。

### 它解决什么问题

* 作为 B→A 的当日流转载体
* 让 A 可以查看、打回、追溯某一次整份上送
* 承载 B 的当日备注

### 最小字段

* id
* 所属 B
* 上送日期
* 备注
* 状态
* 提交时间

### 当前边界

V1 明确：

* 一个 B 一天只按一份上送实例处理
* A 打回的是这整份
* B 重提的也是这整份
* 不做同日多份并列实例管理

---

## 5. 上送实例-日报关联

### 对象定位

记录某个 B 的某日上送实例，实际包含了哪些任务日报。

### 谁产生

系统在 B 发起/提交上送时生成或更新。

### 它解决什么问题

* 还原当次实际上送了哪些日报
* 支撑 A 查看详情
* 支撑缺口计算
* 支撑归档追溯
* 防止只靠临时查询推断

### 最小字段

* id
* 上送实例 id
* 日报 id

### 当前边界

这层关系是实现必需层。
它不一定单独作为业务表达重点，但编码时必须存在。

---

## 二、对象关系

### 1. 总单 → 任务

一个总单下，可由多个 B 创建多条任务。

当前实现里先不额外拆“总单发给 B 的承接对象”，先按最小闭环处理。
也就是：

* 总单是来源
* 任务直接挂总单和 B

### 2. 任务 → 任务日报

一个任务可跨多天，因此可对应多条任务日报。

关系是：
**1 个任务 : 多个日报**

### 3. B 当日上送实例 → 任务日报

一个上送实例包含多条日报。
一条日报在 V1 当前口径下，只进入其所属 B、所属日期的那次上送。

关系实现上按：
**1 个上送实例 : 多条日报关联**

---

## 三、关键规则

---

## 规则 1：任务是主业务对象

B 层主组织单元按任务，不按人。

含义：

* 流程主线围绕任务走
* 人是责任属性，不是主组织轴
* 页面、接口、统计的第一视角都先围绕任务

---

## 规则 2：任务跨天存在

任务不是默认一天结束，而是带开始日期、结束日期的跨天对象。

含义：

* 后续每天是否应报，不靠人工临时判断
* 由任务时间范围自动计算

---

## 规则 3：日报是任务下的单日单实例

C 在某一天针对某个任务，只允许有一条日报实例。

含义：

* 当天第一次填报时创建
* 被打回后在原记录上修改
* 不允许同一天并列多条日报

---

## 规则 4：日报提交后锁定

C 提交日报后，该日报立即锁定。

含义：

* C 不能自己反复修改已提交日报
* 只有 B 打回后才恢复可编辑
* 重提后再次锁定

---

## 规则 5：B 无逐条通过，只有逐条打回

B 对日报的逐条处理，不设计“通过”。

含义：

* 正常项默认成立
* 异常项才显式打回
* 不增加 B 的逐条操作负担

---

## 规则 6：上送自动纳入有效日报

B 发起当日上送时，系统自动带上当天所有：

**已提交且未被打回的日报。**

含义：

* B 不需要逐条勾选正常项
* 上送内容由系统按规则汇总

---

## 规则 7：上送实例必须保留日报集合

B 的当日上送，不只是一个“提交动作”，而必须保留当次到底送了哪些日报。

含义：

* 不能只靠事后按条件再查
* 必须显式落关联
* 打回、归档、缺口、追溯都基于这个集合

---

## 规则 8：逐级打回，不越级改

A 只能打回 B 的当日上送。
B 只能打回 C 的任务日报。

含义：

* A 不直接改 C 日报
* C 不直接越级向 A 修正
* 修正路径始终按层级返回

---

## 规则 9：缺口不单独建对象

缺口不是单独维护的一份业务内容。

口径是：

**缺口 = 应报范围 - 实际上送集合**

含义：

* 不新增“缺口表”
* 缺口由任务时间范围和上送关联反推

---

## 规则 10：V1 不提前引入复杂例外

以下规则当前不进入对象设计：

* 免报
* 暂停
* 恢复
* 迟报复杂口径
* 一任务多人
* 一人多任务冲突

含义：
**对象设计必须严守当前边界，不为未来复杂场景提前加重。**

---

## 四、最小状态

### 任务

* 进行中
* 已结束

### 任务日报

* 未报
* 已提交
* 被打回

### B 当日上送实例

* 未上送
* 已上送
* 被打回

---

## 五、当前冻结结论

这轮先冻结 5 件事：

1. V1 主业务对象是 **任务**
2. 任务是 **跨天对象**
3. 日报是 **任务下的单日单实例**
4. B→A 的上送必须有 **独立上送实例**
5. 上送实例与日报之间必须有 **显式关联**

这 5 条不改，后面就可以往表结构和接口走。

---

4. **状态判定**

**状态 B：对象已稳定，可进入实现层细化。**

也就是现在已经适合往下压：

* 最小表结构
* 页面-动作-接口对应清单

但还不该重新打开复杂边界。

---

5. **保留分支 / 当前不跟**

当前不跟：

* 复杂权限对象
* 复杂日志对象
* 复杂附件对象
* 免报/迟报等例外对象
* 人视角补充主轴

这些都先保留，不进入 V1 对象基线。
