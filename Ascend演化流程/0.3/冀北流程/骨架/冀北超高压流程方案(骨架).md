# 冀北超高压流程方案（当前梳理版）

## 1. 方案目标

本方案面向当前客户现场流程，目标是：

* 尽量贴合现有人工工作方式
* 下级少操作，减少额外负担
* 上级能快速掌握整体情况
* 关键责任、关键缺口可追溯
* 保留复杂性，但不让当前方案过重

当前方案优先解决：

* 主业务对象是什么
* A/B/C 三层怎么协作
* 每天上报与多天任务是什么关系
* 锁定、打回、上送怎么成立
* 统计与缺口口径怎么来

---

## 2. 组织层级定义

为方便讨论，统一使用 A / B / C 三层：

### A：生产管控层

* 角色：管理员
* 职责：下发总单、查看各 B 整体上报结果、必要时打回 B

### B：各分部

* 角色：分部人员
* 职责：承接 A 的总单、下发任务给 C、接收 C 每日上报、发现问题时打回、每天向 A 上报本分部结果

### C：现场工作人员

* 角色：到岗到位人员、工作负责人等
* 职责：围绕分配到的任务，按天填报并提交

---

## 3. 方案核心原则

### 3.1 主组织单元先按任务，不按人

当前 B 层默认按**任务**组织，不按**人**组织。

原因：

* 任务才是流程真正流转的对象
* 以后如果出现“一任务多人”或“一人多任务”，按人做主轴会明显变复杂
* 责任可以挂在人身上，但业务对象仍然应是任务

所以当前方案里：

* **任务是主业务对象**
* **人是责任属性，不是当前主组织轴**

---

### 3.2 任务是跨天对象，日报是任务下的当日记录

当前场景以巡线为例，任务通常不是一天结束，而是持续多天。

所以关系先定义为：

* 一个任务可跨多天存在
* 每一天围绕该任务产生一次当日填报
* 单次任务只是“时间范围刚好只有一天”的特例

即：

**1 个任务 → 多天任务日报**

---

### 3.3 默认纳入，异常打回

B 对 C 的处理不应设计成复杂审核状态机，而应尽量贴合现场真实操作。

当前规则冻结为：

* C 提交后即锁定
* B 没有逐条“通过”按钮
* B 只在发现问题时“打回”
* 没被打回的已报记录，默认可纳入当日上送

也就是：

**正常项不额外处理，异常项才处理。**

---

### 3.4 逐级锁定，逐级打回

这套业务不是自由在线协同编辑，而是强层级闭环。

规则如下：

* C 上送给 B 后，若 B 不打回，则该记录视为成立并锁定
* B 上送给 A 后，若 A 不打回，则该次上送视为成立并锁定
* 错误修正通过逐级打回完成
* 不允许越级直接修改

即：

* A 只能打回 B
* B 只能打回 C
* C 若有问题，需联系 B
* 若 B 认为需要继续向上处理，再联系 A
* 再由 A 打回 B，B 打回 C

---

## 4. 核心业务对象

当前方案中，核心对象分为 4 层。

---

### 4.1 总单

由 A 发起，面向多个 B。

特点：

* 是顶层管理对象
* 负责表达本次工作安排的大框架
* 一个总单可下发到多个 B
* A 主要围绕总单进行整体掌握与督办

---

### 4.2 任务

由 B 基于总单下发给 C。

特点：

* 是当前方案中的主业务对象
* 任务责任落实到具体人
* 任务可跨多天存在
* 后续每天的填报都挂在任务下面
* 当前 B 层默认按任务处理

补充说明：

* 当前虽有“责任到人”的业务要求，但“人”不作为主对象
* 当前先认定：任务是主线，人是任务责任人

---

### 4.3 任务日报

由 C 围绕任务，在某一天提交的一次当日记录。

特点：

* 属于任务下的当日实例
* 一天一次
* 当前先按“单日单实例”理解
* 提交后立即锁定
* 若 B 打回，则该日报实例退回可编辑
* C 修改后再次提交，仍然是这同一份实例继续流转，不另起新条

---

### 4.4 B 当日上送实例

B 在某一天向 A 的一次轻量上送实例。

这个对象要特别说明：

* 它**不是另一套新的重内容对象**
* B 本质上不是重新生产一套日报
* B 更多是在把当天 C 的有效记录往 A 上送
* 最多带少量附件、备注等补充信息

但它又 **必须作为一个独立实例存在** ，因为：

* A 打回的是整份
* B 重提的也是整份
* 它需要记录当次实际上送的 C 记录集合
* 它需要承载时间、上送人、附件、备注、打回关系等信息

所以更准确的说法是：

**B 的上送对象是一个轻量流转实例，不是重内容对象。**

---

## 5. 主流程方案

---

### 5.1 下发流程

#### A → B

A 下发总单给多个 B。

特点：

* 一对多
* B 的数量不固定
* 每个 B 独立承接本分部范围内的工作

#### B → C

B 基于总单拆出任务，下发给 C。

特点：

* 一对多
* 当前默认任务责任到具体个人
* 当前方案不把“人”作为主组织单元，但责任仍落实到具体人

---

### 5.2 日常上报流程

#### C → B

C 每天围绕任务填报并提交任务日报。

规则：

* 一旦提交，记录即锁定
* 若 B 不打回，C 不能再改
* 若 B 打回，C 才可修改并重新提交

#### B → A

B 每天进行一次当日上送。

规则：

* B 不要求等全量齐备才可上报
* 只要 B 认为当前收到的内容没问题，就可以上报
* 如果 B 明知缺项仍上报，责任由 B 自己承担
* B 上报给 A 时，系统默认带上当日所有“已报且未打回”的 C 记录
* 不需要 B 再逐条点通过
* 不需要 B 再手工勾选正常项

---

## 6. B 对 C 的最小操作模型

为了尽量贴合现场、减少操作负担，B 对 C 的逐条处理动作，当前只保留：

### 唯一显式逐条动作：打回

也就是说：

* C 报上来后，B 看内容
* 有问题就打回
* 没问题就不做逐条操作
* 最终在整份上送 A 时，自然带上这些未打回记录

因此，不设计：

* 逐条“通过”按钮
* 逐条“审核完成”按钮
* 逐条“纳入上送”勾选动作

这符合当前人工流程。

---

## 7. 状态与锁定规则

当前方案不建议把用户操作显式做成很重的状态机，但业务规则可以这样理解：

---

### 7.1 C 侧任务日报规则

* 当天应报但未提交：未报
* 已提交：锁定
* 被 B 打回：退回可编辑
* 修改后再次提交：再次锁定

关键点：

* 不做“通过”按钮
* 不打回即默认视为可上送
* 单日单实例，不做并列多条

---

### 7.2 B 侧当日上送规则

* B 每天一份当日上送实例
* 上送后即锁定
* A 打回后，该实例退回给 B
* B 重新处理后再次上送，仍然是这同一份实例继续流转

关键点：

* A 打回的是整份
* B 再上报的也是整份
* 不做多份并列实例

---

## 8. 应报范围与缺口口径

这是当前方案的关键口径之一。

### 8.1 应报范围

由任务下发时确定的**时间范围**决定。

即：

* 一个任务下发时，已经确定其有效时间范围
* 某一天只要落在这个时间范围内，这一天该任务就进入应报范围

因此：

**今天应报哪些任务，不靠人工临时判断，而由系统基于任务时间范围自动计算。**

---

### 8.2 当日上送集合

B 的当日上送实例，必须显式记录：

**本次实际上送了哪些 C 记录。**

这点必须留痕，原因是：

* 需要还原 B 当时到底带了哪些内容
* A 当时看到的汇总是基于哪些记录算出来的
* 后续追责、打回、归档都依赖这个集合

---

### 8.3 缺口口径

缺口不要求 B 再额外维护一套独立内容，但系统必须可反推。

口径为：

**缺口 = 应报范围 - 实际上送集合**

也就是：

* 系统先知道今天哪些任务本来应报
* 再知道 B 本次实际上送了哪些 C 记录
* 二者相减即可得到缺口

这样做的好处是：

* 减少 B 额外填写负担
* 保留系统追责和统计能力
* 不另造复杂缺口对象

---

## 9. A / B / C 的默认查看视角

---

### 9.1 A 侧默认视角

A 默认不直接进入 C 的细语义。

A 的第一层关注应是 B 维度整体情况，例如：

* 本分部应报多少
* 已报多少
* 未报多少
* 异常情况概览
* B 的特殊备注 / 附件说明

A 如有需要，可以继续下钻：

* 到任务
* 到责任人
* 到具体日报明细

但：

**A 默认不直接站在 C 的过程语境里理解问题。**
A 主要对 B 的上报结果负责。

---

### 9.2 B 侧默认视角

当前冻结为：

**B 默认按任务看。**

B 不是先按人组织，也不是先按页面去想，而是先按任务作为主对象处理：

* 哪些任务今天应报
* 哪些任务已报
* 哪些任务被打回
* 哪些任务已随当日整份上送

这里的人仍然重要，但先作为任务的责任属性存在。

---

### 9.3 C 侧默认视角

C 只需要关心：

* 自己今天有哪些任务应报
* 每个任务今天的日报是否已提交
* 哪些被打回需要重提

C 不需要直接理解 A 的语境。

---

## 10. 备注与附件

当前方案允许 B 在向 A 上送时，附带：

* 备注
* 附件

用途主要是：

* 特殊原因说明
* 客观情况说明
* 公司制度或现场限制说明

但要注意：

* B 的备注不是重新生产一套核心业务内容
* B 的内容主线仍然是把 C 的当日有效记录往上送

---

## 11. 当前冻结的关键结论

下面这些是当前已经比较稳定、可以作为方案骨架冻结的结论：

1. A / B / C 三层关系明确
2. A 发总单给多个 B
3. B 基于总单下发任务给 C
4. 当前 B 层主组织单元先按任务，不按人
5. 任务是跨天持续对象
6. 每天填报是任务下的当日记录
7. C 提交后即锁定
8. B 对 C 没有“通过”动作，只有“打回”动作
9. B 上报 A 时，默认带上当天所有已报且未打回的记录
10. B 不要求等全量齐了才可上报
11. 若 B 明知缺项仍上报，责任由 B 承担
12. A 打回 B，B 打回 C，逐级打回，不越级改
13. B→A 存在一个轻量当日上送实例
14. 该实例必须记录本次实际上送的 C 记录集合
15. 应报范围由任务时间范围决定
16. 缺口由系统基于应报范围与实际上送集合反推
17. A 默认看 B 的结果，不默认直接理解 C 的细过程状态

---
