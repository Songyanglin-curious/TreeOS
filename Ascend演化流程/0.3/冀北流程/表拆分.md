3. **局部展开点**

# 冀北超高压流程 V1 最小表结构草案

## 1）总单表 `main_order`

### 作用

A 创建的顶层工作安排。

### 必须字段

* `id`：主键
* `title`：总单标题
* `content`：总单内容说明
* `created_by`：A 创建人 id
* `created_at`：创建时间
* `start_date`：开始日期
* `end_date`：结束日期

### 先不加

* 总单复杂状态
* 多级审批字段
* 复杂统计字段

---

## 2）任务表 `task`

### 作用

B 基于总单创建并派给 C 的主业务对象。

### 必须字段

* `id`：主键
* `main_order_id`：所属总单 id
* `b_user_id`：所属 B 人员 id
* `c_user_id`：责任 C 人员 id
* `task_name`：任务名称
* `start_date`：开始日期
* `end_date`：结束日期
* `status`：任务状态
* `created_by`：创建人 id
* `created_at`：创建时间

### 状态建议

* `IN_PROGRESS`：进行中
* `FINISHED`：已结束

### 说明

这里先不拆“总单发给 B 的承接表”。
当前 V1 先靠 `main_order_id + b_user_id` 落住来源关系。

---

## 3）任务日报表 `task_report`

### 作用

C 针对某任务、某一天提交的一条日报实例。

### 必须字段

* `id`：主键
* `task_id`：所属任务 id
* `report_date`：日报所属日期
* `c_user_id`：填报人 id
* `content`：日报内容
* `status`：日报状态
* `submitted_at`：提交时间
* `rejected_reason`：打回原因
* `updated_at`：最后更新时间
* `start_time`：干活开始时间
* `end_time`：干活结束时间

### 状态建议

* `SUBMITTED`：已提交
* `REJECTED`：被打回

### 关键约束

必须加唯一约束：

* `unique(task_id, report_date)`

含义：

**一个任务在某一天只能有一条日报。**

### 说明

“未报”不是表里必须存一行的数据状态。
未报可以通过：

**应报任务存在，但日报记录不存在**

来判断。

所以表里不需要强行存 `UNREPORTED`。

---

## 4）B 当日上送表 `b_daily_submission`

### 作用

B 某一天向 A 的一次整份上送实例。

### 必须字段

* `id`：主键
* `b_user_id`：所属 B 人员 id
* `submission_date`：上送日期
* `remark`：备注
* `status`：上送状态
* `submitted_at`：提交时间
* `rejected_reason`：A 打回原因
* `updated_at`：最后更新时间

### 状态建议

* `SUBMITTED`：已上送
* `REJECTED`：被打回

### 关键约束

建议加唯一约束：

* `unique(b_user_id, submission_date)`

含义：

**一个 B 在某一天只保留一份上送实例。**

这符合你们当前冻结口径。

---

## 5）上送-日报关联表 `b_daily_submission_report_rel`

### 作用

记录某次 B 当日上送，实际带了哪些日报。

### 必须字段

* `id`：主键
* `submission_id`：上送实例 id
* `task_report_id`：日报 id

### 关键约束

建议加唯一约束：

* `unique(submission_id, task_report_id)`

含义：

**同一份上送里，同一条日报只关联一次。**

### 说明

这张表不能省。
因为你已经冻结了：

**上送实例必须保留“当次实际上送了哪些日报”的显式集合。**

---

# 四、当前最小字段原则

这轮字段只保留四类：

### 1. 主键

每张表都有 `id`

### 2. 关联键

例如：

* `main_order_id`
* `task_id`
* `submission_id`

### 3. 主业务字段

例如：

* `title`
* `task_name`
* `content`
* `remark`

### 4. 必要状态与时间

例如：

* `status`
* `created_at`
* `submitted_at`
* `updated_at`

---

# 五、当前不进表的内容

以下内容当前先不进独立表，也不强塞字段：

* 附件
* 复杂日志
* 缺口结果
* 消息通知
* 免报/暂停/恢复
* 迟报复杂处理
* 一任务多人
* 一人多任务冲突处理
* 复杂权限控制字段
* 多层级审批记录

原因只有一个：

**这些都不是 V1 主流程闭环的最小落库条件。**

# 六、当前最小查询是怎么成立的

### A 看 B 汇总

基于：

* 任务表
* 日报表
* 上送表
* 上送关联表

算：

* 某 B 某天应报多少
* 实际上送多少
* 缺口多少

### B 看任务和日报情况

基于：

* 任务表
* 日报表

看：

* 今天哪些任务应报
* 哪些已报
* 哪些被打回

### C 看自己该报什么

基于：

* 任务表
* 日报表

看：

* 今天我有哪些任务落在时间范围内
* 哪些已有日报
* 哪些被打回

# 七、当前冻结结论

这轮先冻结 4 条：

1. **V1 最小只建 5 张表**
2. **日报表必须有 `(task_id, report_date)` 唯一约束**
3. **上送表必须有 `(b_user_id, submission_date)` 唯一约束**
4. **上送-日报关联表必须显式存在，不能省**

这 4 条不改，后面接口就能定。
