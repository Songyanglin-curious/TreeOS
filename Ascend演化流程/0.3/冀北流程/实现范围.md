# 冀北超高压流程 V1 实现冻结稿

## 一、V1 实现目标

V1 只做主流程闭环，不做复杂边界。

目标只有一个：

**让 A→B→C 下发、C→B→A 每日回送、逐级打回、基础查看 这条主线真实跑通。**

---

## 二、V1 只做这些

### 1. 下发主流程

* A 创建总单
* A 将总单发给多个 B
* B 基于总单创建任务
* B 将任务派给某个 C

### 2. 日报主流程

* C 每天围绕任务填报一条任务日报
* C 提交后锁定
* B 可查看 C 的日报情况
* B 发现问题时可打回日报
* C 被打回后可修改并重新提交

### 3. 上送主流程

* B 每天向 A 发起一次当日上送
* 上送自动带上当天已报且未打回的日报
* B 可补充备注
* A 可查看 B 的当日上送结果
* A 发现问题时可打回 B 的当日上送
* B 被打回后可重新上送

### 4. 基础查看

* A 看各 B 的汇总情况
* A 必要时查看某个 B 的当日上送详情
* B 看任务和日报情况
* C 看自己今天该报什么、哪些被打回

---

## 三、V1 明确不做

以下内容本版全部不进实现：

* 免报 / 暂停 / 恢复
* 迟报、跨天补报的复杂规则
* 一任务多人
* 一人多任务冲突处理
* 特别复杂的权限体系
* 复杂统计报表
* 消息通知
* 很细的附件管理
* 很复杂的操作历史展示
* 越级处理
* 多套审批流
* 复杂状态机扩展
* 各种例外口径的兼容

### 边界原则

**凡是不属于主流程闭环必需项，V1 一律不做。**

---

## 四、V1 核心对象

### 1. 总单

A 发起的顶层工作安排。

最小必需信息：

* id
* 标题
* 内容说明
* 发起人 A
* 创建时间

---

### 2. 任务

B 基于总单创建并派给 C 的主业务对象。

最小必需信息：

* id
* 所属总单 id
* 所属 B
* 责任 C
* 任务名称
* 开始日期
* 结束日期

---

### 3. 任务日报

C 针对某任务、某一天提交的一条日报实例。

最小必需信息：

* id
* 所属任务 id
* 所属日期
* 填报人 C
* 日报内容
* 当前状态
* 打回原因

---

### 4. B 当日上送实例

B 某一天向 A 的一次当日上送。

最小必需信息：

* id
* 所属 B
* 上送日期
* 备注
* 当前状态

---

## 五、实现必需补充

### 1. 上送关联必须显式存在

虽然业务骨架只保留 4 个核心对象，但实现上必须显式记录：

**某个 B 的某日上送实例，实际包含了哪些任务日报。**

这层关系不能靠临时查询推断。

实现上至少要满足：

* 一个上送实例能关联多条任务日报
* 后续查询时能准确还原“当次到底送了哪些日报”
* A 查看详情、缺口计算、归档追溯都基于这层关联

这层可以理解为：
**上送实例-日报关联关系**

它不一定要在业务表达里上升为第 5 个正式业务对象，但在编码实现里必须存在。

---

## 六、V1 关键规则

### 规则 1：应报范围规则

某天某任务是否应报，只由任务开始日期、结束日期决定。

即：
**某日落在任务时间范围内，则该任务在该日进入应报范围。**

---

### 规则 2：日报唯一规则

C 在某一天针对某个任务，只保留一条日报实例。

即：
**单任务 + 单日期 = 单实例。**

被打回后修改的是原实例，不另起新条。

---

### 规则 3：日报锁定规则

C 提交日报后，该日报立即锁定。
只有被 B 打回后，C 才可再次编辑。
C 修改后再次提交，日报重新锁定。

---

### 规则 4：B 最小处理规则

B 对 C 的逐条处理，只有“打回”，没有“通过”。

即：

* 有问题才打回
* 没打回就默认可纳入上送
* 不做逐条通过
* 不做逐条审核完成

---

### 规则 5：自动纳入上送规则

B 发起当日上送时，系统自动带上当天所有：

**已提交且未被打回的任务日报。**

B 不需要逐条勾选正常项。

---

### 规则 6：逐级打回规则

A 只能打回 B 的当日上送。
B 只能打回 C 的任务日报。
不允许越级直接修改。

---

## 七、V1 最小状态

### 任务

* 进行中
* 已结束

### 任务日报

* 未报
* 已提交
* 被打回

### B 当日上送实例

* 未上送
* 已上送
* 被打回

---

## 八、编码起跑条件

只要下面 3 份最小清单不再改，就直接开始编码：

### 1. V1 范围清单

只回答：

* 做什么
* 不做什么

### 2. 对象与规则清单

只回答：

* 有哪些对象
* 规则怎么跑
* 上送关联如何保留

### 3. 页面-动作-接口对应清单

只回答：

* 每个角色最少有哪些页面
* 每个页面能做什么动作
* 每个动作对应哪个接口

**满足这 3 条，就不再继续讨论方案，直接开工。**

---

## 九、当前禁止重新打开的讨论

在 V1 编码启动前，不再展开以下问题：

* 更细状态机要不要补
* 复杂统计要不要先做
* 更细权限要不要加
* 异常边界要不要一起做
* B 是否再按人补一层视图
* 页面是否再优化
* 交互是否再精细化
* 附件和日志是否做完整体系

原则只有一个：

**不影响主流程闭环的，一律不重开。**

---

## 十、V1 最小成功标准

只要下面这条链能完整跑通，V1 就成立：

1. A 创建总单并发给多个 B
2. B 基于总单给 C 创建任务
3. C 在某天提交任务日报
4. B 可以查看并打回日报
5. B 在当天发起一次上送
6. 系统自动带上有效日报
7. A 可以查看并打回上送
8. B 可以重新上送
