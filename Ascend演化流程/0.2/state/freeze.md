

# 1-大框架梳理

当前按 **/freeze（crystallize-freeze.md）** 执行。

## 当前冻结判断

当前节点可以先冻结为：

> **这套流程的主组织方式，已经稳定为“底层单条、上层整份”。**
> **C→B 按单条记录流转与审核；B→A 按整份上报、整份打回、整份重报。**
> **系统设计原则是：下级最少操作完成工作，上级清晰掌握下级状态。**

这不是终局方案，但已经足够作为下一轮继续的稳定基线。

## 当前定位

当前推进节点是：**冀北超高压现场流程的主流程与状态/动作骨架定义。**

当前问题表述可冻结为：

* 层级：A 生产管控层，B 分部，C 现场执行层
* 下发：**A→B→C**
* 回送：**C→B→A**
* 关注重点： **每天完成情况** ，同时兼顾**整个任务周期情况**

当前主线焦点已经押在：

> **先把流程骨架、状态清单、按钮动作清单压稳，再谈权限、页面、边界条件。**

## 本轮关键变化

本轮真正改变后续推进的变化有这些：

1. **“取回”分支被删除**
   不再走“取回后重报”，而改成 **逐级打回** 。
2. **两层业务粒度已经分开**
   * **C→B：单条提交、单条审核、单条打回**
   * **B→A：整份上报、整份打回、整份重报**
3. **B 的上报门槛已经定死**
   * B 不是等全量完备才可报
   * **B 觉得可以报就报**
   * 哪怕有缺口、哪怕一条没有，也可以上报
   * 完整性责任由 B 承担
4. **冻结规则已经稳定**
   * **C 提交后冻结**
   * 只有 **B 打回** 才可改
   * **B 上报后冻结**
   * 只有 **A 打回** 才可重报
5. **A 的角色边界已经清楚**
   * A 默认看 **B 的整体状态与统计视图**
   * A 可下钻到 C 明细
   * 但 **A 一般不直接管 C**
6. **状态与按钮的设计原则已稳定**
   * **C：最少状态、最少按钮**
   * **B：既看单条审核，也看整份上报**
   * **A：只看管理状态与统计结果**

## 已冻结的状态骨架

### C 层（单条记录）

状态：

* 待提交
* 已提交
* 已打回

动作：

* 保存
* 提交
* 查看

### B 层（看 C 单条）

状态：

* 待审核
* 已通过
* 已打回

动作：

* 查看
* 通过
* 打回

### B 层（对 A 整份）

状态：

* 未上报
* 已上报
* 已打回待重报

动作：

* 上报
* 查看本次上报
* 查看打回原因

### A 层（按 B 看）

状态：

* 未上报
* 已上报
* 已打回待重报

动作：

* 查看
* 打回
* 下钻查看 C 明细

## 当前不跟的分支

### 取回机制

当前已被客户新口径替换为“逐级打回”，本轮不再继续跟。

### A 直接处理 C

当前不跟。现阶段结构已经明确：A 主要面对 B 整体，不直接处理 C 单条。
